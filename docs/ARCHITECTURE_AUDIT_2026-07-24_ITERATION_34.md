# Architecture Audit · 2026-07-24 · Iteration 34

## 范围

本轮继续审计多世界、大存档目录与自愈式权威存档恢复的组合路径，重点检查大量世界同时损坏时，主菜单是否仍具有固定成本边界。

## 发现

### 1. 单世界正确不等于多世界可扩展

上一轮已经保证一个损坏世界可以从 `.tmp` / `.bak` 恢复并原子重建 primary，但 `list_worlds()` 会对每个缺失目录的世界立即执行完整修复。

当 50 或 100 个世界同时损坏时：

```text
打开存档浏览器
→ 顺序验证所有候选
→ 同步修复所有损坏世界
→ 每个世界执行恢复暂存、主文件位移和原子提升
→ UI 刷新耗时随损坏世界数线性增长
```

这是明显的主菜单卡顿风险。

### 2. 简单截断会隐藏世界

若只处理前八个目录并跳过其余世界，玩家会误以为存档丢失。恢复预算只能限制高成本写入，不能限制候选读取和世界列表可见性。

### 3. 文件系统顺序不稳定

直接遍历 `DirAccess.get_directories()` 会让每轮进入预算的世界依赖平台返回顺序。长期运行可能出现某些世界反复被排在预算外，无法确定性收敛。

## 决策

### 每次扫描固定八个主文件提升

新增 `MAX_PRIMARY_REPAIRS_PER_LIST := 8`。每轮只允许八次恢复提升尝试，且诊断记录实际使用量。

### 所有有效世界始终可见

预算外世界仍从有效 temporary 或 backup 读取 metadata，立即渲染存档行。它们携带 `recovery_deferred=true`，但不写入新目录。

### 稳定排序与渐进收敛

扫描前按 world ID 排序。已修复世界下一轮从 sidecar 命中，不再消耗预算，因此 20 个损坏世界稳定按 8、8、4 三轮完成。

### 玩家选择优先

`load_world()` 继续使用 `repair_primary=true`。完整加载不受目录预算限制，避免玩家点击待修复世界后还需刷新多次。

### UI 与 F3 可观测

存档浏览器显示待渐进修复数量与每次预算；F3 世界目录行显示待修复数量，并保持 warning。真正的 primary 提升失败仍保持 critical。

## 测试设计

- 20 个世界各保存两代有效数据；
- 删除 catalog 并写入可解析但语义损坏的 primary；
- 第一次扫描验证所有世界可见、修复 8、待修复 12；
- 第二次修复 8、待修复 4；
- 第三次修复 4；
- 第四次 20/20 sidecar 命中；
- 每个 primary 只修复一次；
- 真实桌面保存浏览器展示 20 行与待修复提示；
- 最终仍需通过总 Runtime、桌面矩阵和 Windows Release。

## 结论

恢复路径从“单个世界正确”推进为“多世界成本有界”。没有新增 Timer、后台线程或平行存档领域，权威数据仍由 `world.json` 所有，`catalog.json` 仍是可丢弃派生数据。
