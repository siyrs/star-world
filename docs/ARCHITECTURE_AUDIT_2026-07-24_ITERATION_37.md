# Architecture Audit · 2026-07-24 · Iteration 37

## 范围

本轮继续审计大存档目录的渐进恢复路径。前一轮已经限制每次完整解析 32 个 `world.json`，并使用占位行保证所有世界始终可见。本轮检查读取预算与 16 项 sidecar 写入预算之间是否仍存在跨刷新重复工作。

## 发现

### 1. 读取有界，但相同世界仍会重复解析

读取预算为 32，目录写入预算为 16。一次扫描解析 32 个健康世界后，只能为前 16 个写入 `catalog.json`。后 16 个虽然已经得到准确 metadata，结果没有被保留。

下一次刷新会再次解析这 16 个完整世界。

96 世界旧序列为：

```text
读取：32 → 32 → 32 → 32 → 32 → 16 → 0
总计：176 次
不同世界：96 个
重复读取：80 次
```

这不是预算越界，但仍会重复执行 JSON 解析、深复制、迁移和白名单派生。

### 2. 简单把读取预算降为 16 会降低首屏信息完成速度

把读取预算直接改成与写入预算相同，可以避免重复读取，但会让 96 世界的 metadata 补齐速度减半，并丢失读取与写入成本独立调节的能力。

### 3. 保存完整 payload 作为跨刷新缓存不可接受

把 `world.json` Dictionary 留在内存会复制玩家、机器、农业、动物、探索与方块覆盖，形成第二个大存档驻留域，并增加失效与所有权风险。

需要保留的是可丢弃的严格白名单目录，不是完整世界状态。

### 4. 暂存必须能识别外部变化

世界可能被显式保存、恢复提升、测试工具或外部进程修改。仅按 world ID 命中会展示旧 metadata，甚至把旧 entry 写入新 primary 的 sidecar。

### 5. 统计重置不应改变系统行为

`reset_catalog_diagnostics()` 用于测试和 F3 会话统计。若它顺便清空暂存，观察动作本身会重新引入重复读取。

## 决策

### 最多 64 项瞬时目录暂存

新增：

```text
MAX_STAGED_CATALOG_ENTRIES := 64
```

暂存只保存 `WorldCatalogPolicy.normalize_entry()` 结果、权威字节数和修改时间。它不进入存档，不创建 Timer、线程或新的生命周期参与者。

### 固定读取优先级

```text
catalog sidecar
→ 瞬时目录暂存
→ 完整权威读取
→ 占位 metadata
```

只有写入预算或暂存容量至少有一个可用时，才执行新的完整读取。

### 失效与生命周期

- primary 字节数或修改时间变化：失效并重新读取；
- 显式保存：新 sidecar 成功后清除暂存，sidecar 失败时保存新严格 entry；
- fallback 提升：先清除旧暂存；
- 删除世界：移除暂存；
- 世界目录消失：扫描开始时剪枝；
- 诊断重置：保留暂存，只清零计数。

### 可观测性

存档浏览器显示目录待写、暂存数量和本轮暂存命中。F3 投影只暴露标量，不暴露 entry 内容。

## 真实验收设计

### 40 世界失效回归

- 首轮读取 32、写入 16、暂存 16、占位 8；
- 显式保存一个暂存世界，只移除该项；
- 外部修改另一个暂存世界，使字节数变化；
- 下一轮必须记录恰好一次失效，重新读取更新后的 metadata；
- 健康报告不得包含世界名称或 `block_overrides`。

### 96 世界规模回归

新序列必须为：

```text
读取：32 → 32 → 32 → 0 → 0 → 0 → 0
总计：96
暂存峰值：48 / 64
```

相较旧实现减少 **80 次**完整读取。

### 真实桌面与发行

正式 `GameScene` 创建 40 个世界，验证存档浏览器、F3、准确 metadata、目录待写提示和稳态零读取。最终候选继续通过总 Runtime、三轮 lifecycle soak、完整真实桌面输入/UI 矩阵与 Windows Release。

## 结论

目录恢复现在不仅每次工作有硬预算，也避免把已完成的完整读取在后续刷新中重复执行。`world.json` 仍是唯一权威来源，`catalog.json` 与瞬时暂存都只是可丢弃、可验证、严格白名单的派生投影。
