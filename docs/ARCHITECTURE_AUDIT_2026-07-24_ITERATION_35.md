# Architecture Audit · 2026-07-24 · Iteration 35

## 范围

本轮继续审计多世界、大存档目录的同步刷新路径。上一轮已经限制主文件原子修复次数，本轮重点验证健康 primary 缺失目录 sidecar 时是否仍存在无界写盘。

## 发现

### 1. primary 修复有界，但目录写入仍无界

`SaveService.list_worlds()` 在 catalog miss 后读取权威世界，并在 `primary_ready` 时立即调用 `_write_catalog_entry()`。

当 100 个健康世界的 `catalog.json` 同时缺失时：

```text
目录扫描
→ 100 次权威 fallback
→ 同步重建所有目录
→ 100 次小型原子写入
```

主文件修复预算为 8 并不能限制这条路径，因为所有 primary 都已经健康。

### 2. 复用 primary 预算不合理

若目录重建和主文件提升共用八个槽位，会把成本差异很大的操作混为一谈，也会让已经修复的世界迟迟没有 sidecar。需要独立预算与独立诊断。

### 3. 简单停止处理会隐藏世界

和主文件渐进恢复相同，预算不能限制列表可见性。所有可读取世界必须立即显示，仅推迟派生目录写入。

### 4. 旧健康报告缺少目录写入积压

F3 只能看到主文件待修复数量，无法区分：

```text
primary 尚未健康
与
primary 健康但 catalog 尚未重建
```

运维判断和性能定位因此不完整。

## 决策

### 独立目录写入预算

新增：

```text
MAX_CATALOG_REBUILDS_PER_LIST := 16
```

每次 `list_worlds()` 最多尝试 16 次 `_write_catalog_entry()`。写入失败同样消耗本轮槽位，避免故障磁盘造成无界重试。

### 保持所有世界可见

预算外的健康世界继续返回白名单 metadata，并设置：

```text
catalog_rebuild_deferred=true
```

下一次扫描通过稳定世界 ID 顺序继续收敛。

### 权威主文件零修改

目录-only 压力测试会在扫描前保存每个 `world.json` 文本，最终逐一比较，确保 sidecar 重建不修改权威主文件，也不进入 backup recovery。

### UI 与 F3

存档浏览器显示待建目录数量和每次 16 的预算。F3 将主文件待修复与目录待重建分开显示，目录积压保持 warning，写入失败保持现有错误证据。

## 真实验收设计

Headless 使用 48 个健康世界，验证：

```text
16 → 16 → 16 → 48/48 命中
```

真实 Windows 桌面使用 24 个世界，首轮必须：

```text
24 行全部可见
目录重建 16
待建目录 8
主文件修复 0
```

并通过真实 F3 显示目录写入预算和积压，第二轮完成剩余 8 个，第三轮进入纯 sidecar 命中。

## 结论

目录恢复现在拥有两层不同成本的硬边界：primary 提升最多 8 次，catalog 写入最多 16 次。系统没有新增 Timer、线程、队列存档或第二权威来源；`world.json` 继续唯一拥有世界状态。
