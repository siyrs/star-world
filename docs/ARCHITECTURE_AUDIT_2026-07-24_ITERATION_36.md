# Architecture Audit · 2026-07-24 · Iteration 36

## 范围

本轮继续审计大存档目录的同步主菜单刷新路径。前两轮已经限制 primary 修复和 `catalog.json` 写入，本轮重点检查缺少 sidecar 时的完整权威存档读取是否仍然无界。

## 发现

### 1. 写入有界不等于读取有界

`SaveService.list_worlds()` 在 catalog miss 后会调用 `_read_world_result()`。即使一次只写 16 个 sidecar，旧实现仍会同步解析所有缺失目录的完整 `world.json`。

当 100 个健康世界全部缺失 sidecar 时：

```text
目录扫描
→ 100 次 catalog miss
→ 同步解析所有完整存档
→ 只写入其中 16 个 sidecar
```

主菜单写盘成本已经有界，但主菜单读取成本仍随世界数量和每个世界的 JSON 体积线性增长。

### 2. 复用 sidecar 写入预算会重复做无效工作

若允许读取任意数量世界但只写 16 个目录，预算外世界下一次刷新会再次解析完整 JSON。大存档可能在多个刷新周期中被重复读取。

完整 JSON 读取和小型 sidecar 写入应拥有独立预算，并且读取预算应高于或等于写入预算，保证每次至少有一批读取结果可以进入稳态缓存。

### 3. 简单停止扫描会隐藏世界

不能因为完整读取预算耗尽就停止遍历目录。玩家必须看到所有世界，包括尚未解析真实 metadata 的世界。

因此需要固定大小占位行，使用目录名、候选字节数和明确的 `authoritative_read_deferred=true`，而不是隐藏世界或伪造已经验证的 metadata。

### 4. 玩家主动加载不能受列表预算限制

列表预算只保护主菜单批量扫描。玩家点击“继续”是明确选择，必须立即执行完整加载、恢复和迁移，不能要求先反复刷新列表。

### 5. 健康报告需要区分三类积压

F3 之前已经区分主文件待修复和待建目录。本轮还需要增加待读世界：

```text
primary 尚未健康
primary 健康但完整 metadata 尚未读取
真实 metadata 已读取但 sidecar 尚未写入
```

三种状态成本和风险不同，不能合并为一个模糊“目录异常”。

## 决策

### 独立完整存档读取预算

新增：

```text
MAX_AUTHORITATIVE_READS_PER_LIST := 32
```

每次 `list_worlds()` 最多进入 32 次 `_read_world_result()`。预算在解析前消耗，避免异常数据或失败读取形成无界重试。

### 全部世界使用可操作占位行

预算外世界继续出现在列表中，显示世界目录名、可用候选字节数和“世界信息待读取”。占位行保留真实 world ID，玩家可以直接点击继续。

### 三预算保持独立

```text
primary 修复：8
完整 JSON 读取：32
catalog 写入：16
```

健康世界读取不得消耗 primary 修复预算；未完整读取的世界不得写 sidecar；已完整读取但写入预算耗尽的世界只推迟派生目录。

### 权威数据零修改

96 世界 Headless 会保存每个 `world.json` 的完整文本，在所有读取和目录收敛后逐字节比较。完整读取、占位展示和 sidecar 写入不得改变权威世界。

### UI 与 F3

存档浏览器状态栏显示待读世界数量和每次 32 的预算；占位世界行显示“世界信息待读取”。F3 显示主文件修复、权威读取和目录写入三层积压与预算。

## 真实验收设计

Headless 使用 96 个健康世界，全部删除 sidecar，验证：

```text
读取：32 → 32 → 32 → 32 → 32 → 16 → 0
写入：16 → 16 → 16 → 16 → 16 → 16 → 0
占位：64 → 48 → 32 → 16 → 0 → 0 → 0
```

真实 Windows 桌面使用 40 个世界。首轮必须：

```text
40 行全部可见
完整读取 32
占位 8
sidecar 写入 16
primary 修复 0
```

并通过真实 F3 显示待读世界与权威读取预算，后续刷新补齐 metadata、完成 sidecar，最终进入纯目录命中。

固定候选还必须通过总 Runtime、三轮 lifecycle soak、完整真实桌面输入/UI 矩阵和 Windows Release 实际导出与启动。

## 结论

世界目录现在拥有三层不同成本的同步边界：主文件修复最多 8、完整存档读取最多 32、sidecar 写入最多 16。系统不新增线程、Timer、持久队列或第二权威来源；`world.json` 继续唯一拥有世界状态。
