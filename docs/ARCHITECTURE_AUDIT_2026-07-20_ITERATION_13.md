## 总体结论

项目已经完成 Husbandry、Ranch 和 Exploration 四个功能参与者的显式组合，生态也支持五只敌对、条件精英和可躲避前摇。当前新的主要风险不再是单个服务缺少上限，而是多个有上限的操作在同一帧被重复调用。

原危险环境扫描每次最多读取 125 个方块，但五只敌对同步生成、取消或清除时，每个 `ecology_changed` 都会独立触发完整评估。这个实现“单次有界、总量无界于事件数”，不适合继续增加第二只精英或更密集的战斗生态。

本轮以多敌对危险事件合并为切入点，完成：

```text
原始领域事件保留
→ 同帧原因集合聚合
→ 每帧最多一次危险评估
→ 同方块环境样本复用
→ 多敌对前摇全局可读提示
→ 多死亡/卸载/掉落生产验收
```

## 发现与处置

### P0 · 继续守护

#### 1. 旧世界、数值和 ID 必须不变

本轮没有修改：

- 世界 Seed 和地形；
- 方块或物品 ID；
- 敌对伤害、前摇、范围和冷却；
- 生物生态权重和上限；
- 探矿、Husbandry、Ranch 或奖励 Schema；
- managed animal ID；
- 玩家伤害保护；
- Windows 发行入口。

新增数据全部是瞬时只读诊断，不进入存档。

#### 2. 事件合并不得吞掉领域结果

危险刷新合并只能减少重复评估和重复表现，不能合并或删除：

- `creature_spawned`；
- `died`；
- `ItemPickup`；
- 攻击前摇状态；
- 伤害；
- 背包写入；
- 世界保存。

真实桌面验收会同时检查五只敌人、五个红圈、三次死亡、三个物理掉落和一次保存恢复。

### P1 · 已完成优化

#### 3. 同帧每个生态事件都进行完整危险扫描

旧路径：

```text
phase_changed    → refresh_now
ecology_changed → refresh_now
ecology_changed → refresh_now
ecology_changed → refresh_now
```

每个 `refresh_now()` 最多读取 125 个方块。五只同步生成和五只同步进入攻击状态，可能让同一玩家位置被重复评估十次。

处置：新增 `DangerRefreshBatchPolicy` 和帧尾批次。三类触发原因进入同一个有界集合，批次最多接受 64 个事件，每帧只提交一次危险评估。

诊断明确分开：

```text
immediate_event_count
immediate_refresh_count
coalesced_danger_event_count
dropped_danger_event_count
max_events_in_refresh_batch
```

#### 4. 事件变化不需要重复读取静态环境

昼夜、敌对数量或攻击状态变化时，玩家通常仍在同一个方块。岩浆和洞穴样本在几个微秒内不会因为敌人开始蓄力而改变。

处置：`ExplorationDangerService.refresh_for_events()` 在同一玩家方块复用最近环境样本，但每次重新查询实时的昼夜、生态、敌对和前摇数据。

周期评估继续每 0.75 秒进行新环境扫描，玩家跨方块也会立即失效缓存。

#### 5. 多敌对前摇只能逐只观察

红色实体预警圈适合精确判断单个敌人，但当敌人在玩家身后或多个红圈重叠时，玩家无法迅速理解总攻击数量和最紧迫的一次命中时间。

处置：Spawner 聚合最多 64 个敌对节点中的活跃前摇；危险 Snapshot 和 HUD 显示：

```text
来袭攻击 ×N
精英 ×M
最快 X.X 秒
```

不暴露攻击者坐标，也不改变实体预警圈和真实命中逻辑。

#### 6. 自然死亡没有统一威胁失效事件

Spawner 原本只有显式 `spawn_creature` 和维护卸载路径。生物通过自己的死亡 Tween 离开场景树时，危险服务只能等待下一次生态轮询。

处置：Spawner 监听：

```text
attack_state_changed
died
tree_exiting
```

已死亡实体通过 `is_combat_target_available()` 立即从危险查询中排除。

#### 7. 种群清理和世界清理混用同一入口

Spawner 同时承载生物和 `ItemPickup`。跨世界清理必须删除旧生物和旧掉落，但运行中的种群重置若调用同一入口，会把刚由敌人死亡生成、尚未被玩家拾取的掉落一起删除。

处置：明确拆分两种语义：

```text
clear_creatures()
→ 完整世界生命周期清理
→ 删除生物与全部旧世界瞬时子节点

clear_creature_population()
→ 运行时种群清理
→ 只删除 creatures 组
→ 保留已经生成的 ItemPickup
```

真实测试会让三只敌人同帧死亡、两只敌人由运行时种群清理卸载，然后检查三个物理掉落仍存在、可拾取、可保存和可恢复；返回菜单和进入新世界仍使用完整清理，防止旧世界掉落泄漏。

### P1 · 下一阶段建议

#### 8. 多敌对焦点提示仍只有单目标

全局 HUD 解决了攻击数量和最快时间，但中心上下文提示仍只显示 RayCast 当前目标。

下一轮可增加纯只读优先级策略：

```text
当前瞄准的蓄力精英
→ 当前瞄准的普通蓄力敌人
→ 最近即将命中的敌人
→ 普通目标信息
```

不得自动切换玩家准星或攻击目标。

#### 9. 第二只地图精英应建立在稳定预算上

完成本轮后，可以为另一张地图增加少量精英，但仍需满足：

- 地图限定；
- 低频条件；
- 独立上限；
- 可观察攻击差异；
- 真实反制；
- 有用途掉落；
- 多敌对合并门禁继续成功。

建议优先极寒冰原或荒漠遗迹，不立即引入 Boss、复杂状态效果或大型行为树。

### P2 · 后续优化

#### 10. Machine Base

Furnace 已有成熟的输入、燃料、进度、阻塞、离线推进和位置型保存。下一步应提取小型能力合同，而不是扩大 FurnaceService 为通用 Manager。

#### 11. 建筑连接形状

门、栅栏、梯子和玻璃板连接仍是下一个建造表达里程碑。必须让视觉、碰撞、放置预览和保存使用同一连接/方向合同。

#### 12. CI 组合化

当前专项工作流质量强，但重复安装 Godot 和严格导入。后续可提取 reusable workflow：

```text
strict import
→ static validators
→ domain scripts
→ optional desktop script
→ artifact upload
```

不得以减少运行覆盖为代价降低成本。
