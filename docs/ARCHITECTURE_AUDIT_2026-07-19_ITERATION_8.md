## 总体结论

项目已经具备较强的领域分离、数据合同和真实发行门禁。资源分布、生态危险、探矿成长、原子奖励和 ServiceHub 参与者均已形成可持续基础。

当前最明显的玩家体验缺口位于敌对近战：生态系统能够生成危险，但实际敌对攻击仍是进入范围后瞬时扣血。危险信息和攻击行为之间缺少可读、可躲避的连接。

因此本轮没有立即增加更多敌对种类或精英数值，而是先建立通用、可测试的攻击前摇合同。

## P1 · 本轮发现与处置

### 1. 敌对攻击是瞬时命中

#### 现状

`BaseCreature._choose_direction()` 中：

```text
distance <= attack_range
→ _attempt_attack()
→ take_damage()
```

#### 影响

- 玩家无法观察攻击；
- 不能通过移动躲避；
- 击退和硬直没有打断语义；
- 生态危险提高只表现为更难避免的扣血；
- 后续精英敌人只能通过数值变大制造难度。

#### 处置

引入：

```text
idle → windup → cooldown
```

前摇结束时重新验证目标，允许 `target_evaded` 和 `interrupted`。

### 2. 攻击节奏硬编码在通用生物基类

#### 现状

```gdscript
const HOSTILE_ATTACK_INTERVAL := 5.0
```

#### 影响

- 新敌对物种无法配置独立节奏；
- 基类承担内容配置；
- 数据测试无法验证攻击节奏；
- 玩家受伤冷却与生物攻击频率可能漂移。

#### 处置

新增：

```text
hostile_attacks.json
→ HostileAttackRegistry
→ CreatureFactory
→ BaseCreature
```

静态合同把生物攻击冷却和玩家重复敌对伤害冷却交叉验证。

### 3. 攻击来源固定写成 zombie

#### 现状

通用 `BaseCreature` 调用：

```gdscript
take_damage(attack_damage, "zombie")
```

#### 影响

- 新敌对生物死亡原因错误；
- 伤害来源过滤和诊断无法扩展；
- 基类知道具体物种。

#### 处置

攻击档案提供稳定 `source_id`，基类只消费能力数据。

### 4. 生产与后备僵尸伤害不一致

#### 现状

```text
creatures.json → damage 1
zombie.gd fallback → damage 3
```

正常工厂路径使用 1，但后备路径会突然变为 3。

#### 处置

后备配置改为 1，并增加静态一致性检查。

### 5. 目标移动和被击退没有取消状态

#### 现状

旧攻击只有“冷却是否结束”，没有正在攻击的事务。

#### 影响

- 玩家后退仍可能被瞬时命中；
- 击退敌人不能阻止攻击；
- 硬直与 AI 攻击没有交互；
- 无法诊断为什么攻击没有命中。

#### 处置

稳定取消原因：

```text
target_unavailable
target_evaded
interrupted
```

取消后进入短恢复，不会立刻重新蓄力刷屏。

### 6. 敌对攻击没有视觉和上下文反馈

#### 现状

HUD 能显示区域危险，但不能说明眼前僵尸何时出手。

#### 处置

- 红色发光脉冲预警圈；
- 无碰撞、无射线阻挡；
- 实体焦点读取无坐标攻击 Snapshot；
- 提示说明剩余时间、攻击打断和后退躲避。

## P1 · 建议下一轮

### 1. 少量地图精英生态

前摇合同稳定后，可以安全增加少量精英变体，但必须遵循：

- 每张地图最多一到两种高信号变体；
- 改变攻击模式或环境互动，而不是只增加生命；
- 每种攻击拥有前摇和可读范围；
- 掉落进入现有地图材料或装备成长路线；
- 不立即增加 Boss 和复杂行为树。

建议先实现一个深渊精英僵尸：

```text
较长前摇
→ 较大预警范围
→ 可躲避重击
→ 地图材料相关掉落
```

### 2. 探矿与危险生命周期参与者

日志和奖励已经迁移为参与者；下一批适合迁移：

```text
ExplorationDangerService
ProspectingService
```

约束：

- 保留公共字段和节点路径；
- 不改变探索存档 v3；
- `begin → attach → activate → clear` 顺序有回归；
- 失败启动和返回菜单必须清理；
- 不把功能逻辑塞进 Coordinator。

### 3. Machine Base

熔炉已成熟，但后续机器若继续复制槽位、进度、阻塞和离线推进，会形成多套实现。

建议提取小型能力：

- 输入与输出端口；
- 进度和阻塞；
- 燃料/能源；
- 暂停与有界离线推进；
- 位置型序列化；
- UI Snapshot。

### 4. 建筑交互

优先：

- 门开关和方向状态；
- 栅栏自动连接；
- 梯子攀爬；
- 玻璃板连接；
- 同一形状合同驱动视觉、碰撞、预览和提交。

## P2 · 结构债务

### 1. 生物基础类继续增长

本轮状态机仍位于 `BaseCreature`，因为所有现有敌对生物共用一个基类和同一物理循环。但后续出现远程、范围或多段攻击后，不应继续扩大单一类。

建议演进：

```text
HostileAttackPolicy（已完成）
→ AttackCapability 接口
→ MeleeWindupCapability
→ future RangedCapability
```

只有第二种真实攻击模型出现时才拆能力节点，避免提前抽象。

### 2. CreatureFactory 数据校验仍分散

本轮敌对攻击拥有独立 Registry，但普通生物基础属性仍由 Factory 直接读取。后续应建立统一生物目录诊断，覆盖：

- 生物实现脚本；
- 基础属性；
- 生态档案；
- 敌对攻击档案；
- 掉落物品；
- 牧场能力。

### 3. Actions 前置重复

专项工作流继续重复 Godot 安装和严格导入。可在功能稳定后提取 reusable workflow，但不能牺牲专项可读性和独立 Artifact。
