# 星的世界 · Product & Architecture Roadmap

## 产品定位

《星的世界》不是简单的体素 Demo，而是长期可扩展的单人沙盒游戏基础。

核心原则：

- 玩家体验优先；
- 系统模块化；
- 数据驱动扩展；
- 每个玩法拥有独立领域边界；
- 所有重要功能必须可测试、可保存、可恢复；
- 功能先形成可玩的闭环，再扩展内容数量；
- 真实桌面与最终 Windows Release 是合入主分支的必要证据。

## 已完成里程碑

### 基础运行与发行可靠性

- 真实 WASD、鼠标、按钮和 InputContext；
- 世界启动保护与非空白画面门禁；
- 渐进区块加载和自适应预算；
- F3 诊断与多轮生命周期 soak；
- Windows Release 实际导出、启动、截图、日志和资源退出检查。

### 方块交互与机器

- 工作台世界授权；
- 位置型箱子及内容保护；
- 熔炉原料、燃料、产出、进度和离线恢复；
- 交互、机器、UI 和存档领域分离。

### 玩家体验基础

- 持久化新手引导；
- 上下文操作提示；
- 有界即时反馈；
- Design Token 与 1024×576 布局门禁。

### 工具与资源系统

已完成第一阶段 Minecraft 风格采集闭环：

- 镐、斧、剑工具类型；
- 木、石、铁、钻石工具等级；
- 方块硬度；
- 按住采集与进度状态机；
- 推荐工具和错误工具速度；
- 铁矿、金矿、钻石矿掉落门槛；
- 石头到圆石等特殊掉落；
- 工具耐久、损坏和存档恢复；
- 背包满时保护世界方块；
- 箱子和熔炉的二次提交保护；
- HUD、快捷栏、Tooltip 和真实桌面采集验收。

架构合同见 [TOOL_HARVEST.md](TOOL_HARVEST.md)。

## 下一阶段重点

### 1. 装备与战斗属性

目标：把现有武器和耐久底座扩展为可理解、可比较的角色成长系统。

建议领域结构：

```text
EquipmentService
├── EquipmentSlotPolicy
├── AttributeResolver
├── DamagePolicy
├── DefensePolicy
└── DurabilityAdapter
```

第一阶段范围：

- 主手、头部、胸部、腿部、脚部槽位；
- 武器伤害由 ItemRegistry 数据驱动；
- 防具减伤与最小伤害规则；
- 装备面板与角色属性摘要；
- 装备耐久；
- 保存、加载和死亡/重生合同；
- 不把属性计算塞进 Creature 或 Player UI。

### 2. 采集工具扩展

在现有采集事务上增加内容，而不改变核心结构：

- 铲：泥土、沙子、雪；
- 锄：耕地转换；
- 剪刀：树叶、羊毛等特殊掉落；
- 工具维修；
- 可选的附魔或品质 metadata；
- 采集命中特效和声音层次；
- 世界方块裂纹或阶段性视觉反馈。

优先保持：数据规则、纯策略、事务提交、体验展示四层分离。

### 3. 农业与食物生产链

目标：建立稳定、可保存的长期生存循环。

方向：

- 耕地状态；
- 种子与作物；
- 生长阶段；
- 光照/水源的简化影响；
- 收获与重新种植；
- 动物繁殖；
- 食物生产与熔炉加工衔接。

建议独立 `AgricultureService` 拥有位置型作物状态，世界只保存方块与稀疏覆盖，不让 UI 或 Player 直接推进生长。

### 4. 探索与资源层级

工具等级已经提供资源门槛，下一步需要把门槛转化为探索动力：

- 更明确的矿物深度分布；
- 不同地图的资源偏好；
- 危险区域与高级材料；
- 可理解的探索反馈；
- 避免仅靠增加矿物数量制造重复劳动。

### 5. 自动化机器基础

在 FurnaceService 基础上扩展：

- Machine Base Contract；
- 输入/输出接口；
- 能源接口；
- 自动化管线预留；
- 机器升级或速度模块；
- 世界可视状态。

新增机器应创建明确领域服务或共享小型基础合同，不把 FurnaceService 扩大成万能 MachineManager。

## 产品优先级

当前建议顺序：

```text
P0  基本功能、输入、保存、发行稳定性            持续守护
P1  装备与战斗属性                              下一里程碑
P1  铲/锄/维修及采集视觉反馈                    与装备并行设计
P2  农业与食物生产链                            形成长期生存循环
P2  探索与资源层级                              增强成长动机
P3  更多机器与自动化                            在闭环稳定后扩展
```

每个里程碑必须具备：

```text
用户目标
→ 可玩闭环
→ 领域边界
→ 数据合同
→ 存档合同
→ 失败反馈
→ Runtime 回归
→ 真实桌面验收
→ Windows Release 验收
```

## 工程质量标准

所有新增系统必须满足：

1. 独立领域服务或明确的纯策略；
2. 数据注册表驱动；
3. 唯一状态所有者；
4. 存档兼容与异常数据规范化；
5. 领域回归测试；
6. 真实桌面交互测试；
7. Windows Release 验收；
8. 日志无脚本错误、解析错误和资源泄漏；
9. UI 不直接修改领域 Dictionary；
10. Player 不承担存档、面板或复杂规则。

## 设计规范

### UI

- 使用统一 Design Token；
- 明确视觉层级；
- 支持最低 1024×576；
- 错误状态必须可理解；
- 世界提示不与阻塞面板叠加；
- 纯展示 HUD 必须鼠标透传；
- 关键结果不能只依靠颜色表达。

### 代码

- 避免 God Object；
- 服务职责单一；
- 通过事件和小型端口降低耦合；
- 优先组合；继承仅用于薄适配层；
- 高成本循环必须有预算和上限；
- 动态调用边界必须显式类型收窄；
- 领域写入必须可以验证成功或失败。

### 产品

- 先解决用户能否理解和完成目标，再增加内容数量；
- 不用隐藏惩罚制造“深度”；
- 工具不足、空间不足、权限不足等失败必须提前解释；
- 新功能不能破坏移动、鼠标、按钮、画面、保存或退出资源；
- 每次主分支更新都应保留可复现的发行证据。

## 长期目标

逐步形成：

- 世界系统；
- 生存系统；
- 建造系统；
- 玩家成长系统；
- 农业生态；
- 自动化系统；
- 探索与挑战系统；

共同组成完整、可持续扩展的《星的世界》沙盒生态。
