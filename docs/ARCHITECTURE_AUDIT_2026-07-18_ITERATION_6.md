5. 功能增长是否继续扩大 ServiceHub 和 CI 重复工作。

## 本轮发现与处置

### P0 · 数据驱动目标必须可达

#### 1. 荒漠“富集”条件超过生产资源上限

生产资源档案中荒漠累计矿物阈值为：

```text
675 / 10000 = 6.75%
```

日志的“富集”门槛为：

```text
7.5%
```

若把荒漠地图印记定义为“发现富集区域”，结构校验会通过，但正常生产分布无法稳定完成目标。

处置：

- 荒漠地图印记接受“可观或富集”；
- 限制在浅层或中层，保持地图主题；
- 新静态门禁把资源累计阈值与日志密度门槛交叉比较；
- 任何地图规则超过生产资源理论上限都会直接失败。

这是本轮最重要的产品正确性优化：测试不能只验证 JSON 类型，还必须验证目标在生产数据下可达。

### P1 · 已完成结构优化

#### 2. 探矿服务只接受一个硬编码物品 ID

旧实现：

```gdscript
if item_id != registry.get_tool_item_id():
    return {"handled": false}
```

影响：

- 无法增加不同能力的探矿工具；
- 地图特有成长只能通过普通奖励表达；
- 新工具容易复制整个服务；
- 预算和能力不能在同一 Registry 中审计。

处置：

```text
ProspectingRegistry schema 2
├─ default base tool
├─ five calibrated profiles
├─ per-tool bounded overrides
└─ required map identity
```

`ProspectingService` 只负责选择档案、地图校验和执行同一有界扫描，不为每张地图创建独立服务。

#### 3. 探索里程碑只能读取全局计数

旧里程碑只支持：

- 记录数量；
- 不同区块；
- 深度；
- 密度；
- 危险；
- 深度种类数量。

它无法表达“在某张地图中完成该地图自己的条件”。

处置：新增纯派生 `profile_rule`：

```text
profile_id
+ optional depth set
+ optional density set
+ optional danger set
+ optional minimum danger score
```

所有非空条件必须同时满足。Policy 仍只读取已保存记录，不重新扫描世界，也不保存第二份状态。

#### 4. 地图 ID 和名称在多个领域重复维护

日志和奖励曾分别维护五地图常量。新增功能继续复制会产生：

- 新地图漏接；
- 名称不一致；
- 不同领域顺序漂移；
- 校验器必须重复硬编码。

处置：新增 `MapProfileCatalog`，统一地图 ID、名称、顺序和合法性。日志、奖励和新跨目录门禁使用该合同。

本轮没有一次性重写所有地图注册表；资源和生态仍保留自己的数据档案，但测试会验证它们与共享目录完全一致。

### P1 · 已完成功能推进

#### 5. 地图差异缺少长期成长回报

此前地图差异包括：

- 地形；
- 资源概率；
- 生物生态；
- 实时危险；
- 首次奖励的普通补给。

这些差异仍没有产生长期保留的地图专属能力。

处置：新增：

```text
地图印记
→ 五种唯一材料
→ 五种校准探矿仪
```

专属材料不占用方块 numeric ID，且每一种都有对应配方，避免死内容。

#### 6. 地图专属工具不能破坏旧探矿兼容

处置：

- 旧 `prospecting_kit` ID 不变；
- 旧配方不变；
- 旧工具仍能在五张地图使用；
- 探索存档继续使用 v3；
- 校准工具只改变当前扫描档案；
- 不修改资源分布、Seed、hash 或 salt；
- 校准诊断不扩大持久字段白名单。

### P1 · 测试基础优化

#### 7. 跨目录完整性需要覆盖“完整成长链”

已有目录完整性主要覆盖方块、物品、视觉、采集和放置。本轮新增专门校验：

```text
MapProfileCatalog
→ profile_rule
→ signature reward
→ unique material item
→ calibrated recipe
→ calibrated prospecting profile
```

任何缺失或错误映射都会在静态阶段失败。

#### 8. 真实验收必须覆盖失败和重载

真实桌面用例不仅验证成功制作，还覆盖：

- 真实危险环境；
- 真实右键基础扫描；
- 真实 J 键和鼠标领取；
- 原子制作；
- 真实右键校准扫描；
- 样本预算；
- 无坐标泄漏；
- 返回菜单；
- 完整世界重载；
- claimed、工具和日志不重复。

## 仍需下一轮处理的结构债务

### 1. ServiceHub 深继承链

当前功能通过多层：

```text
base
→ agriculture
→ character
→ repair
→ husbandry
→ ranch
→ exploration
```

逐层叠加。它保证了兼容，但继续增长会带来：

- 生命周期顺序依赖；
- `super` 调用遗漏风险；
- 难以独立安装或测试功能；
- 字段和常量命名碰撞。

建议下一轮从一个低风险领域开始引入 `FeatureLifecycleParticipant` 或 Feature Installer：

```text
install
begin_world
attach_game
activate
save_into
clear
snapshot_into
```

先让探索奖励或探矿作为第一个组合参与者，不一次性重写成熟继承链。

### 2. UI 扩展注册

虽然 Overlay ID 已统一，但基础 `GameUI` 仍直接知道探索日志。建议后续增加小型扩展面板注册合同，注册：

- overlay ID；
- panel；
- input context；
- 是否暂停模拟；
- open/close callback。

不得把所有 UI 都放进万能 PanelManager。

### 3. GitHub Actions 重复前置

专项工作流持续增长，每套都重复：

- checkout；
- 安装 Godot；
- 严格导入；
- 运行单个脚本；
- 上传证据。

建议提取 reusable workflow，同时保留每个功能的可读专项入口。Windows Release 继续作为单一权威发行任务。

### 4. 内容规模与存档压力

下一阶段应补充：

- 长期探索记录和奖励状态体积报告；
- 96+ 物品与更多配方的 UI 滚动性能；
- 大型农场、牧场和方向化建筑压测；
- 多小时生命周期 soak；
- 数据注册表统一诊断报告。
