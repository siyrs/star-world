# 稳定性与性能基线

本文件定义《星的世界》在继续增加玩法前必须长期保持的发布门槛。新功能不得绕过这些模块，也不得以牺牲输入、存档、帧稳定性或世界生命周期为代价。

## 发布门槛

每次合入 `master` 前必须满足：

1. 主菜单、创建世界、加载世界、暂停、继续、死亡、重生和返回菜单均可完成闭环。
2. WASD、方向键、跳跃、快捷栏、背包、合成、攻击、放置和食用均由统一输入合同驱动。
3. “暂停”必须同时阻断玩家输入和世界模拟；暂停与死亡界面关闭后不得遗留全局暂停状态。
4. 保存结果必须来自真实写入结果。保存失败时不得显示成功，也不得自动退出当前世界。
5. 世界存档使用主文件、临时文件和上一版本备份三层恢复策略。
6. 出生区块同步完成，周边区块按预算渐进构建；单帧不得无上限生成全部视距区块。
7. 远离玩家的生物必须被回收，掉落物和生物不得跨世界残留。
8. GitHub Actions 中的全部 Godot 运行时回归必须通过。

## 模块边界

### `SimulationPauseService`

唯一负责写入 `SceneTree.paused`。`GameUI` 只发布暂停意图，其他模块不得直接修改全局暂停状态。服务退出场景树时会恢复非暂停状态，避免测试或世界切换后残留。

### `AtomicJsonStore`

负责 JSON 文件持久化：

- 新内容先写入 `.tmp` 并刷新；
- 旧主文件移动为 `.bak`；
- 临时文件原子替换主文件；
- 替换失败时恢复旧主文件；
- 读取顺序为主文件、有效临时文件、备份文件。

`SaveService` 负责世界 ID、Schema 和 signal，不再自行实现文件替换细节。

### `ChunkStreamingScheduler` 与 `VoxelChunk`

- 出生区块同步构建，保证玩家立即拥有地面和碰撞。
- 周边区块拆成“体素生成”和“网格生成”两个阶段。
- 默认每步处理 2048 个体素，单帧最多两个步骤，软预算为 4ms。
- 任一时刻只保留一个渐进构建任务，移动出视距时可取消。
- 所有区块共享同一体素材质，避免重复资源分配。
- 玩家直接修改的已加载区块仍同步更新碰撞，保证交互一致性。

### `CreaturePopulationPolicy`

只负责统计当前 Spawner 的直接子生物以及判断距离回收。`CreatureSpawner` 负责创建、移除和 signal，避免每次刷新扫描整个场景树。

## 默认性能策略

- 新安装默认区块视距为 3；设置中仍可选择 1–5。
- `render_distance + 1` 作为卸载距离，减少边界抖动。
- 区块构建统计通过 `VoxelWorld.get_streaming_stats()` 暴露，后续性能 HUD 和自动调节应读取该合同。
- 暂停和死亡时世界物理停止；背包和合成界面只阻断玩家输入，不暂停单人世界模拟。
- 生物超过 56 米后回收，保证本地种群持续围绕玩家刷新。

## 自动化回归

完整入口：

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\run_all.ps1 -Godot C:\path\to\godot.exe
```

稳定性专项：

```powershell
godot --headless --path . --script res://tests/qa/runtime_stability_regression.gd
```

专项覆盖渐进区块构建、共享材质、备份恢复、存档列表无副作用、真实暂停、真实保存反馈、单次设置持久化、共享服务所有权和远距离生物回收。

## 功能迭代顺序

新玩法按以下顺序进入：

1. 先补公开合同与运行时回归；
2. 再实现最小可玩路径；
3. 验证旧存档迁移与世界退出清理；
4. 验证稳定帧时间和资源上限；
5. 最后扩充内容、数值和视觉表现。

任何未通过基线的功能只进入独立分支，不直接合入 `master`。
